# DS4 Fork: AMD R9700 (gfx1201) 支持 + Unix Domain Socket 本地加速

本文档描述 [fred-chen/ds4](https://github.com/fred-chen/ds4) 相对于原项目 [antirez/ds4](https://github.com/antirez/ds4) 的两个主要改动。

---

## 1. AMD gfx1201 (RDNA 4) 系列显卡支持

### 概述

原项目 ROCm 后端仅支持 `gfx1151`（Strix Halo / Radeon 8060S）。本 fork 新增了对 AMD RDNA 4 架构显卡的原生支持，包括 **AMD AI PRO R9700 (gfx1201)**。

### 改动内容

**涉及文件：** `Makefile`、`rocm/ds4_rocm_runtime.cuh`、`rocm/ds4_rocm_matmul.cuh`、`rocm/ds4_rocm_q8.cuh`

**关键修改：**

1. **Makefile** — 新增构建目标：
   - `make r9700` — 专为 gfx1201 编译的单架构二进制文件
   - `make rocm` — 编译包含 gfx1151 + gfx1201 的 fat binary
   - 默认 `ROCM_ARCH ?= gfx1151,gfx1201`

2. **运行时检测** — `ds4_rocm_runtime.cuh`：
   - 新增 `gfx12_device` 标志，通过 HIP 设备属性 `prop.major >= 12` 自动检测 RDNA 4 设备

3. **WMMA 内核兼容性** — `ds4_rocm_matmul.cuh` 和 `ds4_rocm_q8.cuh`：
   - gfx1201 (RDNA 4) 废弃了旧版 WMMA 内联函数
   - 用 `#if !defined(__gfx1201__)` 守卫，在 gfx1201 上禁用 WMMA 路径
   - 自动回退到 shared-X 矩阵乘法路径，功能完全等价

### 构建方法

```bash
# 方法 1：仅编译 gfx1201（AMD AI PRO R9700）
make r9700

# 方法 2：编译 fat binary（同时支持 gfx1151 和 gfx1201）
make rocm

# 方法 3：沿用原有目标（仅 gfx1151）
make strix-halo
```

### 使用方法

构建完成后的命令行参数与原项目完全一致，无需额外配置：

```bash
./ds4-server -m /path/to/model.gguf --ctx 32768
```

运行时自动检测 GPU 架构并选择最优内核路径。

## 2. Unix Domain Socket (UDS) 本地通信加速

### 概述

当 Coordinator 和 Worker 运行在同一台机器上时，自动启用 Unix Domain Socket 进行进程间通信，大幅降低分布式推理的通信延迟。原有的 TCP/IP 通信路径完整保留，不受任何影响。

### 设计原则

- **并行而非替代**：UDS 与 TCP 同时监听，互不干扰
- **自动切换**：Worker 和 Coordinator 均无需额外参数，自动检测
- **优雅降级**：UDS 不可用时自动回退到 TCP（remote 场景）
- **协议兼容**：帧格式（WORK/RESULT/HELLO）完全不变，零协议改动

### 改动内容

**涉及文件：** `ds4_distributed.c`

**新增函数：**

| 函数 | 说明 |
|------|------|
| `dist_is_localhost(host)` | 检测 host 是否为 localhost（`127.0.0.1`、`::1`、`localhost`） |
| `dist_uds_make_path(buf, len, port)` | 生成 UDS 路径：`/tmp/ds4_uds_<port>.sock` |
| `dist_create_uds_listener(path, ...)` | 创建 UDS 监听 socket，自动清理旧文件 |
| `dist_connect_uds_once(path, ...)` | 连接到 UDS socket，支持重试 |

**Coordinator 端改动：**

- 当 `--listen` 参数为 localhost 或未指定（默认监听所有接口）时，同时创建 UDS listener
- 接受线程使用 `poll()` 同时监听 TCP 和 UDS 两个 socket

**Worker 端改动：**

- 连接 Coordinator 时，若 host 为 localhost，优先通过 UDS 连接
- Worker 的 data listener 同样自动创建 UDS socket
- 接受线程使用 `poll()` 同时监听 TCP 和 UDS
- Worker-to-worker 连接同样支持 UDS

**连接行为：**

- Coordinator 为 localhost 时：Worker 持续重试 UDS 连接，直到成功（不会 fallback 到 TCP）
- Coordinator 为远程地址时：完全使用 TCP，不受影响
- Worker 启动顺序不影响：即使 Coordinator 后启动，Worker 也会等待 UDS socket 就绪

### 使用方法

**无需任何额外配置。** 当启动参数满足以下条件时自动启用：

```bash
# Coordinator（localhost 监听）
./ds4-server --role coordinator --listen localhost 8081 --layers 0:22 ...

# Worker（连接 localhost Coordinator）
./ds4-server --role worker --coordinator localhost 8081 --layers 23:31 ...
```

省略 `--listen` 参数时 Coordinator 默认监听所有接口，同样会启用 UDS。

### 验证 UDS 是否生效

```bash
# 查看 UDS socket 文件
ls -l /tmp/ds4_uds_*.sock

# 查看 UDS 连接
ss -x | grep ds4
```

Worker 启动日志中会显示 UDS 连接路径：

```
ds4: distributed worker: connected to coordinator localhost:/tmp/ds4_uds_8081.sock
```
